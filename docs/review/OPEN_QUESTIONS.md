# Open Questions

Вопросы, реально возникшие при выполнении CLAUDE-PACK-01. Не отвечаю на них
самостоятельно — они требуют решения владельца проекта.

## Обязательные вопросы (пакет CLAUDE-PACK-01)

1. Какая лицензия будет утверждена для репозитория?
   (`LICENSE` пока содержит временную заглушку "License decision pending".)
2. Где будет размещён основной Git-репозиторий (GitHub organization / URL)?
3. Какие GitHub team aliases будут использоваться в `CODEOWNERS`?
   (Сейчас используются placeholder-псевдонимы `@EPD2-CANON-OWNER`,
   `@EPD2-ARCHITECTURE-OWNER`, `@EPD2-INFRASTRUCTURE-OWNER`,
   `@EPD2-FRONTEND-OWNER`, не привязанные к реальным пользователям.)
4. Будет ли монорепо сохранён после запуска пробника, или платформа перейдёт
   к отдельным репозиториям на сервис? (См. `docs/adr/ADR-001-repository-strategy.md` —
   решение принято только для стартовой стадии.)
5. Какая система issue tracking будет канонической? (Репозиторий сейчас
   содержит только GitHub Issue templates, без предположения, что GitHub
   Issues — окончательный инструмент трекинга.)

## Дополнительные вопросы, возникшие при выполнении пакета

6. `CLAUDE-PACK-01` требует Python 3.12, но окружение исполнения по
   умолчанию (`python3`) указывает на Python 3.11; интерпретатор 3.12
   присутствует отдельно (`python3.12` / управляется через `uv`). Repo
   зафиксирован на Python 3.12 через `pyproject.toml` и `uv`, но стоит
   подтвердить, что целевая среда разработки/CI гарантированно предоставляет
   Python 3.12.
7. Раздел 29 канона («Открытые решения до разработки голосования») содержит
   15 вопросов, явно отложенных до пакета Voting. Они не дублируются здесь,
   так как уже зафиксированы в самом каноне
   (`docs/canonical/TZ-00-domain-event-canon.md`, раздел 29), и не требуют
   решения на инфраструктурном этапе.
8. `scripts/check_repository.py` проверяет только _наличие_ обязательных
   путей, а не то, что их содержимое непустое или соответствует ожидаемому.
   Это подтвердилось на практике: в артефакте, полученном с GitHub Actions
   для закрытия PACK-01, пять обязательных governance-файлов
   (`.github/workflows/ci.yml`, `.github/pull_request_template.md`, три
   `.github/ISSUE_TEMPLATE/*.yml`) были обнулены до 0 байт, но проверка
   всё равно сообщала "OK: all 72 required paths are present" — потому что
   путь существовал. Аналогично `npm run format:check` (Prettier) не
   зафиксировал бы обнуление, так как пустой файл тривиально
   соответствует любому стилю форматирования. Рекомендация: рассмотреть
   добавление проверки ненулевого размера (и/или контрольных сумм для
   особо чувствительных файлов, включая канон) в
   `scripts/check_repository.py` или отдельный скрипт — решение
   архитектурное и оставляется на усмотрение владельца проекта, не
   реализовано самостоятельно в рамках этого прохода. Полная хронология
   инцидента — в `docs/handover/PACK-01-REPORT.md`, Revision 4.

## Дополнительные вопросы, возникшие при выполнении CLAUDE-PACK-02

9. `ADR-002` рекомендует будущее добавление канонического события
   `identity.verification_revoked` (minor-версия канона), вместо
   переиспользования `identity.verification_expired` для явного отзыва.
   Не выполнено самостоятельно в рамках этого пакета — решение
   архитектурное и требует отдельного ADR + версии канона.
10. `ADR-004` вводит 21 дополнительный reason code
    (`contracts/reason-codes/pack-02.yml`, `source: pack-02-adr-004`),
    отсутствующих в каноне (раздел 24). Рекомендуется будущий
    minor-версия канона, включающая эти коды в раздел 24 — не выполнено
    самостоятельно в рамках этого пакета.
11. Идемпотентность на уровне команды (CT-00-04) с явным
    caller-supplied `event_id` реализована пока только в
    `epd2_credential_service.application.issue_participation_credential`
    (см. `tests/contract/test_ct00_04_event_idempotency.py`). Аналогичные
    команды в остальных трёх сервисах (`create_account`,
    `start_identity_verification`, `evaluate_eligibility`,
    `create_eligibility_snapshot`) пока принимают только
    entity-уровневую идемпотентность (например, `credential_id`/
    `(eligibility_rule_id, rule_version)` дедупликацию в сторе), но не
    принимают свой собственный `event_id` параметр для дедупликации на
    уровне Audit Core при повторе одного и того же запроса. Не
    заблокировано (Audit Core сам по себе корректно идемпотентен по
    `audit_event_id` — раздел 9.3 пакета), но для полной симметрии между
    сервисами стоит рассмотреть добавление того же параметра везде в
    будущем пакете.
12. Проверка "forbidden central identity-participation mapping table or
    file" (пакет, раздел 15) реализована в
    `scripts/check_forbidden_files.py` как эвристика по имени файла
    (identity/person-термин + participation/credential/account-термин +
    map/link/join-термин одновременно). Она находит статически
    обнаружимый случай, который явно называет пакет, но не может
    обнаружить mapping, выраженный только как рантайм-данные или как
    пара колонок внутри иначе безобидно названного файла — эта граница
    задокументирована в самом скрипте, а не скрыта.
